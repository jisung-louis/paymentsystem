# Payment System :: PolyCube BE Assignment

#### 지원자 전지성

## 프로젝트 진행 과정에서 중요하게 생각한 부분

이번 과제에서 가장 중요하게 본 부분은 **결제 흐름과 할인 정책의 책임을 분리**하는 것이었습니다.

요구사항에 `할인 정책은 수시로 추가/변경 가능하다`, `신규 서비스는 비즈니스 요구사항이 자주 바뀔 수 있다`는 조건이 있었기 때문에, 결제 서비스가 특정 할인계산 방식에 직접 의존하지 않도록 설계했습니다.

`PaymentService`는 주문 조회, 할인 계산 요청, 결제 생성, 저장이라는 결제 흐름만 조율합니다. 실제 할인 조건 판단과 할인 금액 계산은 `DiscountPolicy` 구현체와 `DiscountCalculator`가 담당합니다.

이 구조를 선택한 이유는 새로운 할인정책이 추가되거나 기존 정책이 변경되더라도 결제 서비스의 변경을 최소화하기 위해서입니다.

## main 브랜치에서의 기본 설계

`main` 브랜치에서는 기본 요구사항에 맞춰 회원 등급별 할인 정책을 분리했습니다.

- `VipDiscountPolicy` : VIP 회원에게 1,000원 고정 할인
- `VvipDiscountPolicy` : VVIP 회원에게 주문 금액의 10% 할인
- `DiscountCalculator` : 회원 등급에 맞는 할인 정책을 찾아 할인 결과 계산
- `Payment` : 결제 성공 시 주문, 최종 결제 금액, 결제 수단, 결제 일시 저장

A 요구사항 단계에서는 한 주문에 회원 등급 할인이 하나만 적용되므로, 결제 흐름과 할인 정책을 분리하는 데 집중했습니다.

## feature 브랜치에서 변경한 설계

추가 요구사항(B)에서는 `포인트 결제 시 최종 금액에서 추가 5%를 중복 할인`해야 합니다.

이 요구사항은 기존처럼 할인 정책 하나만 선택하는 구조로는 자연스럽게 처리하기 어렵습니다. 중복 할인이 가능해야 하기 때문입니다.

그래서 `feature` 브랜치에서는 할인 구조를 단일 정책 선택 방식에서 **우선순위** 기반 누적 할인 방식으로 변경했습니다.

변경된 할인 흐름은 다음과 같습니다.

1. 결제에 필요한 정보를 `DiscountContext`로 만든다.
2. 적용 가능한 `DiscountPolicy`를 모두 찾는다.
3. 정책의 `priority` 순서대로 정렬한다.
4. 각 정책을 현재 금액에 차례대로 적용한다.
5. 정책별 할인 결과를 누적한다.
6. 최종 결제 금액과 할인 이력을 결제에 저장한다.

## 할인 정책 구조

`DiscountPolicy`는 다음 책임을 가집니다.

- `policyName` : 결제 이력에 저장할 정책명
- `priority` : 중복 할인 적용 순서
- `isApplicable` : 현재 결제 조건에 정책이 적용 가능한지 판단
- `apply` : 현재 금액에 할인 적용

현재 구현된 정책은 다음과 같습니다.

- `VipDiscountPolicy` : VIP 회원 1,000원 고정 할인, 우선순위 100
- `VvipDiscountPolicy` : VVIP 회원 10% 할인, 우선순위 100
- `PointPaymentDiscountPolicy`: 포인트 결제 시 현재 금액의 5% 추가 할인, 우선순위 200

회원 등급 할인은 먼저 적용되고, 포인트 결제 할인은 등급 할인이 적용된 이후의 금액에 적용됩니다.

예를 들어 VVIP 회원이 20,000원 주문을 포인트로 결제하면 다음과 같이 계산됩니다.

1. VVIP 할인 : 20,000원의 10%인 2,000원 할인
2. 중간 금액 : 18,000원
3. 포인트 결제 할인 : 18,000원의 5%인 900원 할인
4. 최종 결제 금액 : 17,100원

동시에 적용되는 정책의 우선순위가 같으면 계산 순서가 모호해지므로 예외를 발생시키도록 했습니다. 이를 통해 정책 추가 시 의도하지 않은 순서 의존성이 생기지 않도록 했습니다.

## 결제 이력 보존 전략

추가 요구사항에서는 할인 정책이 수정되거나 삭제되어도 과거 결제 데이터와 할인 이력이 보존되어야 합니다.

이를 위해 결제 완료 시점의 데이터를 스냅샷으로 저장하도록 변경했습니다.

`Payment`에는 다음 정보를 저장합니다.

- 결제 당시 상품명
- 결제 당시 주문 원가
- 결제 당시 회원 등급
- 총 할인 금액
- 최종 결제 금액
- 결제 수단
- 결제 상태
- 결제 일시

할인 정책별 적용 결과는 `PaymentDiscountHistory`에 별도로 저장합니다.

`PaymentDiscountHistory`에는 다음 정보를 저장합니다.

- 적용 당시 회원 등급
- 적용 정책명
- 할인 종류
- 할인율
- 할인 금액
- 할인 적용 전 금액
- 할인 적용 후 금액
- 할인 적용 순서

이렇게 저장하면 나중에 `VvipDiscountPolicy`의 할인율이 10%에서 다른 값으로 변경되거나 정책 코드가 삭제되어도, 이미 완료된 결제는 저장된 스냅샷과 할인 이력을 기준으로 조회할 수 있습니다.

과거 결제 데이터를 다시 정책으로 재계산하지 않고, 결제 당시 확정된 결과를 DB에 남기는 방식입니다.

## 테스트

기본 요구사항의 등급별 할인 테스트에 더해, 추가 요구사항을 검증하는 테스트를 작성했습니다.

할인 계산 테스트에서는 다음 내용을 검증하도록 했습ㄴ다.

- NORMAL 회원은 할인이 적용되지 않는다.
- VIP 회원은 1,000원 고정 할인이 적용된다.
- VVIP 회원은 주문 금액의 10% 할인이 적용된다.
- VIP 할인 금액은 주문 금액을 초과하지 않는다.
- 포인트 결제 시 등급 할인 후 남은 금액에서 5% 중복 할인이 적용된다.
- 동시에 적용되는 정책의 우선순위가 같으면 예외가 발생한다.

결제 테스트에서는 다음 내용을 검증했습니다.

- 포인트 결제 성공 시 결제 스냅샷과 할인 이력이 저장된다.
- 정책 변경/삭제로 재계산 결과가 달라져도 과거 결제 데이터와 할인 이력은 보존된다.

## 설계 변경의 논리적 근거

`main` 브랜치의 구조는 기본 요구사항에는 충분했지만, `feature` 브랜치의 중복 할인 요구사항을 반영하기에는 한계가 있었습니다.

특히 할인 정책을 하나만 선택하면 다음 문제가 생깁니다.

- 회원 등급 할인과 결제 수단 할인 같은 '할인 원인'을 정확히 표현하기 힘듦
- 어떤 정책이 먼저 적용되는지 모호함 (`findFirst` 때문에 Spring Bean 주입 순서에 의존됨)

이를 해결하기 위해 `feature` 브랜치에서는 할인 정책을 우선순위 기반으로 누적 적용했습니다.

그리고 각 정책의 계산 결과를 결제 이력으로 저장하도록 개선했습니다.

이 설계는 새로운 할인 정책이 추가되어도 `DiscountPolicy` 구현체를 추가하고 우선순위를 부여하면 기존 결제 흐름을 크게 바꾸지 않고 확장할 수 있습니다.

또한 결제 당시의 결과를 스냅샷으로 저장하기 때문에, 운영 중 정책이 자주 변경되더라도 과거 결제 데이터의 정합성을 유지할 수 있습니다.
