# Payment System :: PolyCube BE Assignment

#### 지원자 전지성

## 프로젝트 진행 과정에서 중요하게 생각한 부분

이번 과제에서 가장 중요하게 본 부분은 결제 흐름과 할인 정책의 책임을 분리하는 것이었습니다.

요구사항에 `할인 정책은 수시로 추가/변경 가능하다`, `신규 서비스는 비즈니스 요구사항이 자주 바뀔 수 있다`는 조건이 있었기 때문에, 결제 서비스가 특정 할인 계산 방식에 직접 의존하지 않도록 설계했습니다.

`PaymentService`는 주문 조회, 할인 계산 요청, 결제 생성, 저장이라는 결제 흐름만 조율합니다. 실제 할인 조건 판단과 할인 금액 계산은 `DiscountPolicy` 구현체와 `DiscountCalculator`가 담당합니다.

이 구조를 선택한 이유는 새로운 할인정책이 추가되거나 기존 정책이 변경되더라도 결제 서비스의 변경을 최소화하기 위해서입니다.

## main 브랜치에서의 기본 설계

`main` 브랜치에서는 기본 요구사항에 맞춰 회원 등급별 할인 정책을 분리했습니다.

- `VipDiscountPolicy`: VIP 회원에게 1,000원 고정 할인
- `VvipDiscountPolicy`: VVIP 회원에게 주문 금액의 10% 할인
- `DiscountCalculator`: 회원 등급에 맞는 할인 정책을 찾아 할인 결과 계산
- `Payment`: 결제 성공 시 주문, 최종 결제 금액, 결제 수단, 결제 일시 저장

이 단계에서는 한 주문에 회원 등급 할인이 하나만 적용되므로, 결제 흐름과 할인 정책을 분리하는 데 집중했습니다.

## 할인 정책 설계

할인 정책은 `DiscountPolicy` interface로 분리했습니다.

각 할인 정책은 자신이 적용 가능한 회원 등급을 판단하고, 해당 정책의 할인 금액을 계산합니다.

`PaymentService`는 VIP 할인인지 VVIP 할인인지 알지 못하고, `DiscountCalculator`에게 할인 계산 책임을 위임하도록 설계했습니다.

이렇게 설계한 이유는 할인 정책이 추가되거나 변경되더라도 결제 흐름 자체는 최대한 유지하기 위해서입니다.

## 결제 설계

결제 요청은 주문 ID와 결제 수단을 입력받습니다.

결제 서비스는 주문을 조회한 뒤, 주문자의 회원 등급과 주문 원가를 기준으로 할인 금액을 계산합니다.

그 후 최종 결제 금액을 계산하고, `Payment` 엔티티를 생성하여 저장합니다.

`Payment`에는 주문 정보, 최종 결제 금액, 결제 수단, 결제 상태, 결제 일시를 저장합니다.

결제 객체 생성 책임은 `PaymentService`에 직접 두지 않고 `Payment` 엔티티의 정적 생성 메서드로 분리했습니다.

## 테스트

테스트는 요구사항에 적힌 회원 등급별 할인 로직을 유닛 테스트로 검증했습니다.

검증한 내용은 다음과 같습니다.

- NORMAL 회원은 할인이 적용되지 않음
- VIP 회원은 1,000원 고정 할인이 적용됨
- VVIP 회원은 주문 금액의 10% 할인이 적용됨
- VIP 할인 금액은 주문 금액을 초과하지 않음 (최종금액 음수 방지 확인)

## feature 브랜치에서 확장할 방향

지금 구현한 A 요구사항 (`main` 브랜치) 에서는 기본 요구사항에 맞춰 등급 할인 정책을 단일 적용하도록 구현했습니다.

하지만 추가 요구사항(B 요구사항, `feature` 브랜치) 에서는 포인트 결제 시 최종 금액에서 추가 5%를 중복 할인해야 합니다.

이 경우 하나의 결제에 회원 등급 할인과 결제 수단 할인이 함께 적용될 수 있으므로, `feature` 브랜치에서는 할인 계산 구조를 우선순위 기반 누적 할인 방식으로 확장할 계획입니다.

또한 추가 요구사항에는 정책 수정/삭제 후에도 과거 결제 데이터와 할인 이력을 보존해야 한다는 조건이 있습니다.

이를 위해 `feature` 브랜치에서는 결제 당시의 주문 정보, 회원 등급, 적용 정책명, 할인율, 할인 금액 등을 스냅샷으로 저장하는 구조로 설계를 개선할 계획입니다.
