## 프로젝트 진행 과정에서 중요하게 생각한 부분

이번 과제에서 가장 중요하게 본 부분은 결제 흐름과 할인 정책의 책임을 분리하는 것이었습니다.

특히 요구사항에 `할인 정책은 수시로 추가/변경이 가능하다`, `신규 서비스는 비즈니스 요구사항이 자주 바뀔 수 있다`는 조건이 있었기때문에, 결제 서비스가 특정 할인계산 방식에 직접 의존하지 않도록 설계했습니다.

`결제` 도메인에 할인 정책의 세부 계산은 `discount`에 위임하고, `PaymentService`는 결제 흐름만 조율하게 하여 추후 할인 정책 제거에도 용이하도록 설계했습니다.

그리고 할인정책을 `DiscountPolicy` interface로 분리했습니다.

`DiscountCalculator`에서 멤버 등급별로 자신이 적용 가능한 조건을 판단하고, 해당 정책의 할인 금액을 계산합니다. 

이 구조를 선택한 이유는 새로운 할인정책이 추가되거나 기존 정책이 변경되더라도 결제 서비스의 변경을 최소화하기 위해서입니다.

이로써 테스트를 하거나 정책 추가/제거/수정에 용이하도록 다른 코드에 영향을 주는 결합도를 최대한 줄였습니다.

그리고 `결제` 서비스 로직에서 `결제 객체 생성`의 책임을 두지 않고 `Payment` 엔티티에 책임을 분리했습니다.

> main(요구사항 A)에서는 기본 요구사항에 맞춰 등급 할인 정책을 단일 적용하도록 구현했습니다. feature(요구사항 B)에서는 포인트 결제 중복 할인이 추가되므로, 할인 계산 구조를 우선순위 기반 누적할인 방식으로 확장할 예정입니다.
